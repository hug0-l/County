# рҹҺҜ Implementation Plan: NTP-based Time Service with Failover

## рҹ“Ң Problem Statement
The current system relies solely on the local system clock and performs only a cosmetic connectivity check to `worldtimeapi.org`. It does **not** actually calibrate its clock against any external source, leaving broadcast timing at the mercy of the PCвҖҷs clock drift.

We need a proper time service that:
- Fetches real time from a configurable NTP/HTTP time server  
- Computes and applies an offset to the local clock  
- Falls back to local time if the server is unreachable  
- Defaults to Hong Kong Observatory time source  
- Shows the time source status clearly in the UI  
- Allows full configuration (server URL, sync interval, manual sync) from the Settings page

## рҹ§  Design Decisions & Constraints
- **Browser runtime**: Cannot use raw NTP (UDP). Must use an HTTP-based time API that returns UTC time in JSON.
- **CORS**: The chosen time service must support Cross-Origin Resource Sharing from `file://` or localhost.
- **Default server**: The user asked for вҖңйҰҷжёҜеӨ©ж–ҮеҸ°вҖқ (Hong Kong Observatory). However, the ObservatoryвҖҷs official time source (`time.hko.hk`) is NTP onlyвҖ”no public CORS-enabled HTTP API exists.  
  вҶ’ We default to `https://worldtimeapi.org/api/timezone/Asia/Hong_Kong` (accurate Hong Kong time, CORS-friendly). The user can change this to any valid time API URL (including future Observatory HTTP endpoints). This is noted in the help.
- **Offset storage**: The last known offset and sync timestamp are persisted in `localStorage` so that a fallback offset survives page reloads.
- **Integration**: The existing `timeOffset` variable (currently set to `1` but unused) will be repurposed. `getCalibratedDate()` will add this offset before timezone conversion.

### Card 3 вҖ“ Core time synchronization function `syncTime(serverUrl)`
**Description**  
Implement an async function that:
1. Fetches the given URL with a 5вҖ‘second timeout
2. Parses the JSON response (support `"datetime"` ISO string, `"unixtime"` seconds, `"utc_datetime"`)
3. Computes `offset = server_utc_epoch - Date.now()`
4. On success: saves `timeOffset`, updates `localStorage` (`ntpLastOffset`, `ntpLastSyncTime`), calls `updateNtpStatusUI()`
5. On failure: logs warning, **preserves** existing offset (no overwrite), calls `updateNtpStatusUI()` with error state
6. Always resolves/rejects so callers can await

**Acceptance Criteria**
- [ ] With a reachable server, the clock drifts to match the server
- [ ] With an unreachable server, the last good offset stays in effect and status shows вҖңвҡ пёҸвҖқ
- [ ] Timeout after 5 seconds triggers error path
- [ ] Invalid JSON or missing keys are handled gracefully

## рҹ”Ғ Verification Strategy
1. **UnitвҖ‘style manual test**: Open the app with a network connection вҶ’ clock should sync and badge show offset.
2. **Offline test**: Disconnect network, reload вҶ’ app uses last stored offset, badge shows вҖңжң¬ең°жҷӮйҗҳвҖқ.
3. **Invalid URL test**: Set a nonвҖ‘existent URL in settings вҶ’ sync fails, old offset kept, error shown.
4. **Timeout test**: Point to a URL that hangs вҶ’ after 5 seconds, sync fails, no crash.
5. **Persistence test**: Sync once, close browser, reopen вҶ’ offset reapplied before first render.
6. **Integration test**: All existing engine functions (matrix, cues) continue to work with the adjusted time.
